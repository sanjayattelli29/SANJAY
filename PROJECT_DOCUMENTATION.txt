PROJECT OVERVIEW
================================================================================
This is an Insurance Platform backend application built using .NET 8 (C#).
It handles insurance policy applications, claims management, user authentication,
and various administrative functions for an insurance company.

The system supports 4 types of users:
1. Admin - Manages the entire system, creates agents and claim officers
2. Customer - Applies for policies and raises insurance claims
3. Agent - Reviews and approves policy applications
4. Claim Officer - Reviews and processes insurance claims


WHAT IS A POCO CLASS?
================================================================================
POCO stands for: "Plain Old CLR Object" (CLR = Common Language Runtime)

Simple Explanation:
- POCO is just a simple C# class with properties (data fields)
- It represents data or business objects (like Customer, Policy, Claim, etc.)
- It has NO complex logic, NO inheritance from framework classes (with exceptions)
- Just plain properties like: Id, Name, Email, etc.
- Also called "Entity Classes" or "Model Classes" or "Domain Models"

Example:
    public class Person 
    {
        public string Name { get; set; }
        public int Age { get; set; }
        public string Email { get; set; }
    }

In simple terms: POCO = A class that holds data, nothing fancy.


PROJECT ARCHITECTURE (CLEAN ARCHITECTURE / LAYERED ARCHITECTURE)
================================================================================
This project follows "Clean Architecture" (also called "Onion Architecture").
It separates code into 4 distinct layers with specific responsibilities.

Think of it like building floors:
- API Layer (Top Floor) = Entry point, receives requests
- Application Layer (Floor 2) = Business rules and logic
- Domain Layer (Floor 3) = Core business models (POCO classes)
- Infrastructure Layer (Ground Floor) = Database, external services


LAYER 1: DOMAIN LAYER (Core Business Models)
================================================================================
Location: Domain/ folder
Purpose: Contains all POCO classes (entities) that represent core business data

What's inside:
- Entities/ folder: POCO classes (database tables)
- Enums/ folder: Constants and fixed values

TOTAL POCO/ENTITY CLASSES: 5 main entities + 10 supporting classes = 15 classes

Main Entity Classes (Database Tables):
--------------------------------------
1. ApplicationUser.cs
   - Represents users in the system (Admin, Customer, Agent, ClaimOfficer)
   - Extends IdentityUser (for authentication)
   - Properties: Id, UserName, Email, FullName, AadhaarNumber, BankAccountNumber
   - Role: Stores all user information

2. PolicyApplication.cs
   - Represents a customer's insurance policy application
   - Properties: Id, UserId, PolicyCategory, TierId, CalculatedPremium, Status,
                 AssignedAgentId, CoverageAmount, StartDate, ExpiryDate, etc.
   - Role: Tracks policy applications from submission to approval

3. InsuranceClaim.cs
   - Represents an insurance claim raised by a customer
   - Properties: Id, PolicyApplicationId, UserId, RequestedAmount, ApprovedAmount,
                 Description, IncidentDate, Status, AssignedClaimOfficerId, etc.
   - Role: Tracks insurance claims from submission to payment

4. ClaimDocument.cs
   - Represents documents uploaded for a claim (medical bills, reports)
   - Properties: Id, ClaimId, FileId, FileName, FileUrl, FileSize, UploadedAt
   - Role: Stores document metadata for claim verification

5. PolicyConfiguration.cs
   - Stores insurance policy configuration (tiers, premiums, risk factors)
   - Contains nested classes for complex configuration
   - Role: Defines available insurance plans and pricing rules

Supporting Classes in PolicyConfiguration.cs:

7. PolicyTier - Different tier levels (Bronze, Silver, Gold, etc.)
8. RiskFactors - Risk calculation factors
9. AgeMultiplier - Age-based risk multipliers
10. ProfessionMultiplier - Profession-based risk multipliers

12. SmokingMultiplier - Smoking habit risk factors

HOW ALL LAYERS WORK TOGETHER (REQUEST FLOW)
================================================================================

Example: Customer Login Process
--------------------------------

1. FRONTEND sends request:
   POST /api/auth/login
   Body: { "email": "customer@test.com", "password": "Pass123" }
   
2. API LAYER (AuthController):
   - Receives HTTP request
   - Extracts JSON body into LoginDto
   - Calls IAuthService.LoginAsync(loginDto)
   
3. APPLICATION LAYER (IAuthService interface):
   - Defines contract: LoginAsync(LoginDto) should return AuthResponseDto
   
4. INFRASTRUCTURE LAYER (AuthService implementation):
   - Implements actual logic:
     * Queries database for user using ApplicationDbContext
     * Verifies password
     * Generates JWT token
     * Returns AuthResponseDto with token and user info
   
5. DOMAIN LAYER (ApplicationUser entity):
   - Represents the user data structure in database
   
6. RESPONSE flows back:
   AuthService → IAuthService → AuthController → HTTP Response → Frontend

This layered approach ensures:
- Separation of concerns (each layer has one job)
- Testability (can test each layer independently)
- Maintainability (changes in one layer don't break others)
- Flexibility (can swap implementations easily)


PROGRAM.CS - APPLICATION CONFIGURATION
================================================================================
Location: API/Program.cs
Purpose: Entry point, configures all services and middleware

Key Configurations:
-------------------
1. Database Connection
   - Uses SQL Server (connection string in appsettings.json)
   - Entity Framework Core for ORM (Object-Relational Mapping)

2. Authentication & Authorization
   - ASP.NET Core Identity (for user management)
   - JWT Bearer Token authentication
   - Password requirements (minimum 6 chars, require digit)

3. Dependency Injection
   - Registers all services for automatic injection
   - AuthService, PolicyService, ClaimService, FileStorageService

4. CORS (Cross-Origin Resource Sharing)
   - Allows frontend from different domain to call APIs

5. Swagger/OpenAPI
   - Auto-generates API documentation
   - Available at: /swagger URL when running

6. JSON Serialization
   - Handles circular references
   - Ignores null values in responses

HOW TO ANSWER COMMON QUESTIONS
================================================================================

Q: How many APIs do you have?
A: 33 API endpoints across 7 controllers

Q: How many POCO classes?
A: 15 classes total (5 main entities: ApplicationUser, PolicyApplication, 
   InsuranceClaim, ClaimDocument, PolicyConfiguration + 10 supporting classes)

Q: What is API folder?
A: The API folder is the entry point layer containing controllers that expose
   HTTP endpoints for client applications to interact with the backend.

Q: What is Application folder?
A: The Application folder contains business logic contracts (interfaces) and
   DTOs. It defines WHAT operations the system can do, not HOW.

Q: What is Domain folder?
A: The Domain folder contains core business models (POCO/Entity classes) that
   represent the fundamental data structures of the insurance business.

Q: What is Infrastructure folder?
A: The Infrastructure folder contains the actual implementation of business
   logic, database access code, and integration with external services.

Q: What is Clean Architecture?
A: Clean Architecture separates code into layers where inner layers (Domain)
   don't depend on outer layers (API), making the code more maintainable,
   testable, and flexible.

Q: How does a request flow through the system?
A: Frontend → API Controller → Interface → Service Implementation → Database
   → Service → Interface → Controller → Frontend

Q: What database are you using?
A: SQL Server with Entity Framework Core ORM

Q: How is authentication handled?
A: JWT (JSON Web Tokens) with ASP.NET Core Identity for user management
